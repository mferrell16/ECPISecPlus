Buffer Overflow Lab

Security+ SY0-701 · 2.3
The Core Concept
A buffer overflow happens when a program writes more data into a fixed-size memory region (a buffer) than it was allocated to hold. The excess data spills into adjacent memory, corrupting it.

Unlike XSS — which is intentional injection — buffer overflow exploits a fundamental mistake: the program trusted the user to send the right amount of data, and didn't check.
Exam anchor: If a question says "input exceeded the allocated space" or "adjacent memory was overwritten" → Buffer Overflow. If it says "attacker placed code into a running process" → Memory Injection (different attack).
THE STACK — WHERE THIS HAPPENS
Programs store local variables and function call information on the stack — a region of memory. One of those pieces of information is the return address: where execution goes when the current function finishes.

A buffer overflow that reaches the return address lets an attacker redirect execution to their own code — called shellcode.

Buffer Overflow vs Memory Injection — Exam Differentiator
BUFFER OVERFLOW
Cause: Bad input exceeds buffer size
How: Adjacent memory corrupted by excess bytes
Target: Stack or heap memory boundary
Look for: "input exceeded allocated space", "program crashed", "adjacent memory"
MEMORY INJECTION
Cause: Deliberate code placement
How: Attacker injects code into a running process
Target: Process memory space (DLL injection, hollowing)
Look for: "injected into process", "DLL injection", "process hollowing"
KEY RULE
Overflow = accidental input-driven corruption.

Injection = attacker intentionally put code in the right place.

If the question says "crashed" or "exceeded" → Overflow.

Building the Exploit Payload
A real buffer overflow exploit has three parts, sent as one continuous string of bytes. Click each component to add it to your payload, then launch the exploit.
1
NOP Sled Landing Pad
A sequence of 0x90 (NOP = No Operation) bytes. The CPU slides through them harmlessly until it hits the shellcode. Gives the attacker a large target — they don't need to know the exact address of the shellcode, just anywhere in the sled.
2
Shellcode Malicious Payload
The actual code the attacker wants to run — often opens a shell, creates a backdoor user, or disables security controls. Injected as raw bytes inside the overflow.
3
Return Address Override The Hijack
The last bytes of the payload overwrite the return address on the stack, pointing execution back into the NOP sled. When the function returns, the CPU jumps to the attacker's code instead.

How to Stop Buffer Overflows
1. Bounds Checking
Use safe functions that enforce length limits. Replace strcpy() with strncpy(). Replace gets() (banned in C11) with fgets(). The fix is simple — the vulnerability exists because developers skipped it.
2. ASLR
Address Space Layout Randomization — the OS randomizes where the stack, heap, and libraries are placed in memory on every run. The attacker can no longer hardcode a return address — they don't know where the shellcode will land.
3. DEP / NX Bit
Data Execution Prevention / No-Execute — marks the stack and heap as non-executable. Even if the attacker injects shellcode, the CPU will refuse to execute bytes in data regions, crashing the program instead.
4. Stack Canaries
A random value ("canary") is placed between the buffer and the return address at function entry. Before returning, the program checks if the canary changed. If so, the stack was corrupted and the program terminates safely instead of jumping to attacker code.
5. Safe Languages
Languages like Rust, Go, and modern Java perform automatic bounds checking on arrays and strings. Buffer overflows are nearly impossible by design — the runtime checks every array access and panics safely rather than writing out-of-bounds.
6. Input Validation
Validate and sanitize all input before processing. Enforce maximum lengths at the application layer. Reject or truncate inputs that exceed expected size — never pass unchecked user data to memory-unsafe functions.
VULNERABLE vs SAFE CODE
❌ Vulnerable — C (classic)
char buffer[4];
strcpy(buffer, userInput); // no limit

char line[64];
gets(line); // banned — no length arg
✓ Safe — bounded C
char buffer[4];
strncpy(buffer, userInput, sizeof(buffer) - 1);
buffer[3] = '\0'; // always null-terminate

char line[64];
fgets(line, sizeof(line), stdin);
❌ Vulnerable — C (with canary disabled)
// gcc -fno-stack-protector -z execstack vuln.c
// ASLR off, DEP off, no canary
// → classic exploit works directly
✓ Safe — hardened compilation
// gcc -fstack-protector-strong (canary)
// -pie -fPIE (ASLR)
// -z noexecstack (DEP/NX)
Exam note: ASLR, DEP/NX, and stack canaries are OS/compiler-level mitigations. They reduce exploitability but don't fix the root cause — unsafe code. Bounds checking fixes the root cause.
