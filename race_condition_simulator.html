The Core Concept
A race condition occurs when a program's behavior depends on the timing or sequence of events — and an attacker can manipulate that timing to their advantage.

The classic form is TOCTOU: Time of Check to Time of Use. The program checks a condition, then uses the result — but an attacker changes the state between the check and the use.

The program trusted its own check — but the world changed underneath it.
Exam anchor: "Time of Check / Time of Use" or "TOCTOU" on any exam question → Race Condition. Look for: a check that happens separately from the action, two processes accessing the same resource, a "window of opportunity."

Why The Name "Race Condition"?
The attacker is literally racing the legitimate process. They need to win the race — get their swap in during the gap — before the program uses the checked value.

The vulnerability is not in the check itself, nor in the use itself — it's in the assumed atomicity between them. The program assumes "what I checked is still true when I act." That assumption is wrong if another process can run in between.
THREE REAL-WORLD EXAMPLES
File System (Classic TOCTOU)
Program checks if /tmp/safefile is safe, then opens it. Attacker swaps it for a symlink to /etc/passwd in the gap. Program now writes to /etc/passwd.
Bank Account (Double-Spend)
Two simultaneous withdrawals. Both threads check balance = $100, both see "sufficient funds," both complete the withdrawal. Account goes to –$100.
Security Check Bypass
Antivirus scans an uploaded file, reports it clean. Between the scan and execution, attacker swaps it for malware. Program runs the malware trusting the AV result.

RACE CONDITION / TOCTOU
Cause: Timing gap between check and use
How: Attacker modifies state during the window
Look for: "TOCTOU", "between check and use", "simultaneous", "window of opportunity", two threads/processes
BUFFER OVERFLOW
Cause: Input exceeds buffer size
How: Adjacent memory corrupted by too many bytes
Look for: "exceeded allocated space", "adjacent memory overwritten", crashes, shellcode

Cause: Unsanitized input rendered as HTML
How: Script runs in other users' browsers

The File Symlink Attack
This is the classic TOCTOU exploit. A privileged program (running as root) checks whether a file is safe before writing to it. An attacker creates a symlink in the gap — the program ends up writing to a sensitive system file.
VULNERABLE PSEUDOCODE
// Privileged program (runs as root)
if (is_safe("/tmp/userfile")) { // ← TIME OF CHECK
  // ... attacker swaps /tmp/userfile here ...
  write_data("/tmp/userfile", data); // ← TIME OF USE
}

How to Stop Race Conditions
1. Atomic Operations
Combine the check and the use into a single, uninterruptible operation. No other thread can run between them. Databases call these transactions — either the whole thing succeeds or none of it does.
2. Mutexes / Locks
A mutex (mutual exclusion lock) ensures only one thread can access a shared resource at a time. While Thread A holds the lock, Thread B must wait. Eliminates the race window entirely for that resource.
3. Open with O_NOFOLLOW
For file-system TOCTOU: open files with the O_NOFOLLOW flag, which refuses to follow symlinks. Or use openat() with a directory file descriptor to lock the path at check time.
4. Minimize the Window
The smaller the gap between check and use, the harder the race. Move the check as close as possible to the use. Don't perform expensive operations between them. Reduces exploitability even if not eliminated.
5. Re-verify at Use Time
Check the condition again immediately before acting. Use file handles (already-opened file descriptors) instead of file paths — a path can be changed; an open file descriptor cannot.
6. Principle of Least Privilege
If the vulnerable process runs with minimal privileges, a successful exploit causes minimal damage. A TOCTOU on a non-privileged process cannot write to /etc/passwd. Don't run things as root unnecessarily.
VULNERABLE vs SAFE CODE
❌ Vulnerable — TOCTOU File Access
// Check and open are TWO separate calls
if (access("/tmp/file", W_OK) == 0) {
  // ← attacker swaps /tmp/file HERE
  fd = open("/tmp/file", O_WRONLY);
  write(fd, data, len); // writes to wrong file!
}
✓ Safe — Atomic open + check
// open() itself does the check atomically
fd = open("/tmp/file",
        O_WRONLY | O_NOFOLLOW);
if (fd >= 0) {
  write(fd, data, len); // safe
}
❌ Vulnerable — Bank Withdrawal (no lock)
// Two threads run this simultaneously
if (balance >= amount) { // both threads: TRUE
  balance -= amount; // both subtract!
} // balance goes negative
✓ Safe — With Mutex Lock
mutex.lock();
// only ONE thread can be here at a time
if (balance >= amount) {
  balance -= amount;
}
mutex.unlock();
Exam note: The fix for race conditions is always about making the check and use atomic — inseparable. Mutexes, transactions, and atomic OS calls all achieve this in different ways.
