WHAT IS DNS CACHE POISONING?

A DNS resolver caches records it has looked up to avoid querying upstream servers on every request. Cache poisoning (aka DNS spoofing) injects forged records into that cache — mapping a legitimate domain to an attacker-controlled IP. Every user who queries the poisoned resolver gets sent to the wrong address.

The classic exploit (pre-DNSSEC) exploited the fact that DNS uses UDP — a connectionless protocol with predictable transaction IDs. An attacker who could guess or brute-force the 16-bit transaction ID could win a race to inject a forged response before the real authoritative server replied.

Kaminsky Attack (2008): Dan Kaminsky discovered that attackers don't need to poison the exact record — they can poison the NS record for the zone itself, redirecting all future lookups for an entire domain. This was a critical vulnerability affecting nearly all resolvers.

WHY IT WORKS — TRANSACTION IDs

DNS queries over UDP include a 16-bit transaction ID (TXID). The resolver accepts whichever response arrives first with a matching TXID. Old resolvers used predictable TXIDs — attackers could flood responses guessing all 65,536 possible values before the real reply arrived.

Old Resolvers (Vulnerable)
Query → TXID: 0x0001
Query → TXID: 0x0002
Query → TXID: 0x0003
// Sequential — trivial to predict
// Attacker floods all 65536 TXIDs
Modern Resolvers (Hardened)
Query → TXID: 0x8f3a
Query → TXID: 0x2c91
Query → TXID: 0xb54e
// Random + source port randomization
// 16-bit × 16-bit = 32-bit entropy
Real-world impact: A poisoned cache can redirect users of an entire ISP or organization to phishing sites, intercept credentials, or silently MitM encrypted traffic if the attacker presents a forged cert.

DEFENSES
DNSSEC
Cryptographically signs DNS records. Resolvers validate signatures — a forged record with no valid signature is rejected regardless of TXID. The gold standard defense.
PRIMARY FIX
Source Port Randomization
Randomizing both the TXID and UDP source port raises the attack entropy to ~32 bits, making brute-force flooding statistically infeasible before TTL expires.
HARDENING
DNS over HTTPS / TLS
DoH/DoT encrypts DNS queries, preventing MITM interception and making off-path cache injection significantly harder.

WHAT IS DNS AMPLIFICATION?

A DNS amplification attack is a volumetric DDoS technique that abuses open DNS resolvers as bandwidth amplifiers. The attacker sends small queries to open resolvers with the source IP spoofed to be the victim's address. The resolvers respond with large answers — sent to the victim.

The amplification factor is the ratio of response size to query size. A 50-byte query for a large ANY record can return a 4,000-byte response — an 80× amplification. With thousands of open resolvers, an attacker with a modest upstream connection can generate hundreds of Gbps of traffic toward the victim.

ANY record abuse: DNS ANY queries ask for all record types for a domain — the response can include A, AAAA, MX, TXT, NS, SOA records in one packet. Attackers use domains with large TXT/DKIM records to maximize response size. RFC 8482 deprecated ANY queries to address this.

DEFENSES
BCP38 — Ingress Filtering
ISPs implement BCP38 to drop packets where the source IP doesn't match the originating network. Eliminates IP spoofing — the prerequisite for reflection/amplification attacks.
ROOT FIX
Rate Limiting (RRL)
DNS Response Rate Limiting caps how many responses a resolver sends to the same source IP per second. Limits the amplification each resolver can contribute to an attack.
MITIGATION
Close Open Resolvers
Configure DNS resolvers to only answer queries from authorized networks (internal clients, not the entire internet). Eliminates the resolver as a weapon.

WHAT IS DNS TUNNELING?

DNS tunneling encodes arbitrary data inside DNS query and response packets — using the DNS protocol as a covert communication channel. Because most firewalls and proxies allow outbound DNS traffic, attackers use tunneling to exfiltrate data or maintain command-and-control (C2) communication even in heavily firewalled environments.

The attacker controls an authoritative nameserver for a domain they own (e.g., evil.com). Data is encoded in subdomains of queries (aGVsbG8.evil.com) — those queries flow through the corporate DNS resolver to the attacker's NS server, which decodes the payload. Responses carry data back the same way.

Common tools: iodine, dnscat2, dns2tcp. Used in red team engagements, APT C2 infrastructure, and data theft from air-gapped or heavily filtered networks. Detectable by behavioral analysis — not packet content.

Security+ exam tips: Know DNSSEC as the primary cache poisoning fix. Know BCP38 (ingress filtering) and closing open resolvers for amplification. For tunneling, the detection vector is behavioral — query length/entropy/volume — not content inspection. All three are considered network-layer attacks that bypass traditional application-layer controls.
