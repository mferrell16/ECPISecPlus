THE CORE CONCEPT
Cross-Site Scripting (XSS) lets an attacker inject malicious JavaScript into a webpage that is then executed inside other users' browsers — not the attacker's.

The key insight: the server trusts user input and reflects it back as HTML without sanitizing it. The browser sees it as <script> and runs it.

Exam tip: XSS targets other users' browsers. If a question says "malicious script executes when another user visits the page" — that's XSS.
THREE TYPES TO KNOW
STORED XSS Most Dangerous
Payload is saved in the database (e.g., a comment). Every user who loads the page runs the script. Persistent — works even after attacker logs off.
Example: Attacker posts a comment containing <script>stealCookies()</script>. Everyone who views comments is compromised.
REFLECTED XSS One-Time
Payload is in a URL. Server reflects it back in the response without storing it. Requires tricking the victim into clicking a crafted link.
DOM-BASED XSS Client-Side
Happens entirely in the browser's DOM — the server never sees the payload. JavaScript reads attacker-controlled input (URL hash, etc.) and writes it directly to the page.
XSS vs SQLi vs CSRF — EXAM DIFFERENTIATOR
XSS
Target: Other users' browsers
Payload: JavaScript in input fields
Goal: Execute code in victim's browser
Look for: <script>, cookie theft, defacement
SQL INJECTION
Target: The database
Payload: SQL syntax in input fields
Goal: Read/modify/delete DB records
Look for: OR 1=1--, SQL keywords
CSRF
Target: The victim's authenticated session
Payload: Hidden requests, forged forms
Goal: Perform action victim didn't intend
Look for: "forged request", state-change

WHAT COOKIES CONTAIN
When a user logs into a website, the server sets a session cookie. This cookie is automatically sent with every request — it's how the server knows "this is Alice, she's already logged in."

XSS can read this cookie via document.cookie and send it to the attacker's server. The attacker can then impersonate the victim without ever knowing their password.
Why it works: JavaScript running in a browser can access document.cookie for the current domain — unless the cookie has the HttpOnly flag set.

WHAT THE ATTACKER CAN DO
Session Hijacking: Paste the stolen session cookie into their own browser. The server sees it as a valid session.
Account Takeover: Transfer funds, change email/password, exfiltrate data — all as the victim.
Lateral Movement: If session is for an admin, full platform compromise.
HOW TO STOP XSS
1. Input Validation + Output Encoding
Never trust user input. Encode all output so <script> becomes &lt;script&gt; — the browser displays it as text, never executes it.
2. Content Security Policy (CSP)
An HTTP header that tells the browser which scripts are allowed to run. Blocks inline scripts and scripts from untrusted origins.
3. HttpOnly Cookie Flag
Marks session cookies so JavaScript cannot access them. Eliminates cookie theft via XSS. The Secure flag prevents transmission over HTTP.
4. Web Application Firewall (WAF)
Sits in front of the application and inspects requests. Can detect and block known XSS patterns before they reach the server.
5. Modern Frameworks
React, Angular, Vue automatically encode values before rendering to the DOM, making stored XSS much harder to introduce accidentally.
6. SameSite Cookie Attribute
Restricts cookies from being sent in cross-site requests. Reduces the impact of CSRF attacks that pair with XSS.
VULNERABLE vs SAFE CODE
❌ Vulnerable
// Directly inserts user input as HTML
element.innerHTML = userComment;

// Attacker posts: <script>stealCookies()</script>
// Script executes in every visitor's browser
✓ Safe
// Sets text content, never parsed as HTML
element.textContent = userComment;

// Or encode before inserting:
element.innerHTML = encode(userComment);
// <script> becomes &lt;script&gt; — safe text
