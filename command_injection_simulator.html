WHAT IS COMMAND INJECTION?

Command Injection occurs when an attacker supplies input to an application that is passed directly to a system shell without proper sanitization. The operating system executes the attacker's commands with the same privileges as the web application process.

Unlike SQL Injection — which targets a database — command injection targets the underlying operating system. A successful attack gives the attacker a foothold on the server itself: they can read files, create users, download malware, or pivot to internal networks.

Impact: Full server compromise, lateral movement, data exfiltration, ransomware deployment, persistent backdoor installation — one of the highest-severity injection types.
● CRITICAL CVSS v3: up to 10.0
HOW IT WORKS

A web app offers a "ping" tool. The developer builds the OS command like this:

server-side code (PHP)
// Developer's intent: ping the user-supplied host $cmd = "ping -c 4 " . $_GET['host']; shell_exec($cmd); // Intended: ping -c 4 google.com // Injected: ping -c 4 google.com; cat /etc/passwd

The shell processes the semicolon as a command separator — both commands run:

user input
→
shell_exec()
→
OS executes both cmds
Key principle: The shell doesn't know where the developer's command ends and the attacker's begins. It sees valid shell syntax and executes all of it.
INJECTION OPERATORS

Shell metacharacters that let attackers chain commands:

;   Run second cmd regardless of first
&&  Run second cmd only if first succeeds
||  Run second cmd only if first fails
|   Pipe output of first into second
`cmd` Backtick: substitute cmd output
$(cmd) Subshell substitution (bash)

HOW TO PREVENT COMMAND INJECTION
1. Avoid Shell Calls Entirely
The most effective defense — never pass user input to a shell. Use language-native libraries instead. Need to ping? Use a Python socket library. Need DNS lookup? Use a resolver API. Remove the shell invocation entirely.
2. Parameterized Commands
If OS calls are unavoidable, use APIs that pass arguments as arrays — not shell strings. This tells the OS to treat input as a literal argument, never parsed by a shell interpreter. Never construct shell strings via concatenation.
3. Input Validation — Allowlist
Define exactly what valid input looks like — then reject everything else. For a ping tool: only allow alphanumeric characters and dots (^[a-zA-Z0-9.]+$). Reject any input containing shell metacharacters.
4. Least Privilege
Run the web application as a low-privilege OS user (e.g. www-data not root). Even if injection succeeds, the attacker's commands inherit minimal system access — limiting damage and making privilege escalation a required extra step.
5. Containerization / Sandboxing
Run application processes in containers (Docker) or sandboxed environments with restricted filesystem access and no outbound network access. An escaped container still can't reach the host or internal network without additional vulnerabilities.
6. WAF + Egress Filtering
A WAF can block known injection signatures at the perimeter. Egress (outbound) firewall rules prevent reverse shell callbacks — even if injection succeeds, the attacker can't phone home. Defense in depth reduces blast radius.
VULNERABLE vs SECURE CODE
❌ Vulnerable — Shell String Concatenation
// Python — user input embedded in shell string
import os

host = request.args.get("host")
result = os.system("ping -c 4 " + host)

# Input: google.com; cat /etc/passwd
# Both commands execute as app user!
✓ Secure — subprocess with args array
// Python — input passed as argument, not shell string
import subprocess, re

host = request.args.get("host")
if not re.match(r'^[a-zA-Z0-9.\-]+$', host):
  abort(400)
result = subprocess.run(
  ["ping", "-c", "4", host], # array!
  shell=False, capture_output=True)
# No shell interpreter invoked — safe
❌ Vulnerable — Node.js exec()
// exec() spawns a shell — dangerous!
const { exec } = require('child_process');

const host = req.query.host;
exec(`ping -c 4 ${host}`, (err, stdout) => {
  res.send(stdout);
});
// Template literal = string concat = injection
✓ Secure — Node.js execFile()
// execFile — NO shell, args as array
const { execFile } = require('child_process');
const validator = require('validator');

const host = req.query.host;
if (!validator.isFQDN(host) && !validator.isIP(host))
  return res.status(400).send('Invalid host');
execFile('ping', ['-c', '4', host], (err, out) => {
  res.send(out);
});
// execFile never invokes a shell
DETECTION INDICATORS

What to look for in logs:

Shell metacharacters in params: ; | & ` $( )
Known recon commands: whoami id uname cat /etc
Outbound connections after injection (reverse shells)
Unusual child processes spawned by the web server
New cron jobs or startup scripts added to server

Security+ exam tips:

Command Injection = Injection category (OWASP A03)
Best fix = avoid shell calls / use safe APIs
Key term: shell metacharacter
Also look for: OS command injection
Testing tools: Burp Suite, commix, manual
Related: RCE (Remote Code Execution)
