WHAT IS CSRF?

Cross-Site Request Forgery (CSRF) tricks an authenticated user's browser into sending an unauthorized request to a site where the user is already logged in. The server sees the request — complete with the victim's valid session cookie — and executes it as if the victim intentionally made it.

The attacker never sees the victim's credentials. They don't need to. They just need the victim to visit a malicious page while logged in to the target. The browser automatically attaches the session cookie to every request for that domain — that's the exploit.

Key insight: CSRF abuses the browser's built-in cookie behavior. Cookies are sent automatically with every matching request — the browser doesn't check whether the request originated from the target site itself.
CSRF vs XSS — CRITICAL DISTINCTION

THE ATTACK FLOW

Three conditions must be true for CSRF to work:

1
Victim has an active session
Alice logs in to bank.com. A session cookie is stored in her browser for bank.com.
2
Victim visits the attacker's page
Alice opens an email link, forum post, or ad that takes her to evil.com while still logged in to bank.com.
3
Browser silently sends forged request
evil.com contains a hidden form or img tag that fires a POST to bank.com/transfer. The browser attaches Alice's session cookie automatically.

CSRF TOKENS — HOW THEY WORK

A CSRF token is a secret, unpredictable value generated by the server and embedded in every legitimate HTML form. When the form is submitted, the server checks that the token matches the one it issued — a cross-site attacker can't read the token from a different origin, so they can't forge a valid request.

The key property: The token is tied to the user's session and changes with each request. An attacker on evil.com cannot read a value embedded in bank.com's HTML — the browser's same-origin policy prevents it.
<form action="/transfer" method="POST">
  <input name="to" type="text">
  <input name="amount" type="text">
  <input type="hidden"
         name="csrf_token"
         value="a8f5f167f44f4964e6c9...">
</form>

OTHER CSRF PROTECTIONS IN ACTION
SameSite Cookie: Strict
Cookie is not sent on cross-site requests at all. The browser checks whether the request originated from the same site as the cookie's domain. If not — the cookie is withheld.
Set-Cookie: sessionid=abc123;
  SameSite=Strict;
  HttpOnly;
  Secure
BLOCKS CSRF
Origin / Referer Header Check
The server inspects the Origin or Referer header on state-changing requests. If the origin is not bank.com — reject. Simple, but can be bypassed if headers are absent.
// Server-side check
if (origin !== 'https://bank.com') {
  return res.status(403);
}
PARTIAL DEFENSE
Double Submit Cookie
The server sends the CSRF token as both a cookie and a form field. On submission, it checks they match. An attacker can't read or set the cookie for bank.com from evil.com.
Cookie: csrf=TOKEN123
Body: csrf_field=TOKEN123
Server: cookie === field?
→ match = accept

HOW TO PREVENT CSRF
1. CSRF Tokens (Synchronizer Token Pattern)
Generate a cryptographically random token per session (or per request). Embed it in every state-changing form as a hidden field. Validate it server-side before processing. This is the most widely recommended primary defense.
2. SameSite Cookie Attribute
Set SameSite=Strict or SameSite=Lax on session cookies. Strict blocks all cross-site sends. Lax allows top-level navigations but blocks subresource requests. Modern browsers default to Lax if unset.
3. Verify Origin / Referer
On all POST/PUT/DELETE/PATCH requests, check the Origin header (preferred) or Referer header. Reject requests from unexpected origins. Useful as a defense-in-depth layer — not sufficient alone.
4. Custom Request Headers (AJAX)
For API endpoints, require a custom header like X-Requested-With: XMLHttpRequest. Simple HTML forms from another origin cannot set custom headers — only JavaScript can, and CORS prevents cross-origin JS requests with credentials.
5. Re-authenticate for Sensitive Actions
For high-impact actions (password change, large transfers, account deletion), prompt the user to re-enter their password or confirm with MFA. Even if a forged request arrives, without re-authentication it won't complete.
6. Avoid GET for State Changes
Never use GET requests to perform state-changing operations (transfers, deletions, updates). GET requests can be triggered by <img>, <link>, and <script> tags from any page — they are trivially CSRF-able.
VULNERABLE vs SECURE CODE
❌ Vulnerable — No CSRF Protection
// Express.js — processes any POST
app.post('/transfer', (req, res) => {
  const { to, amount } = req.body;
  // No token check!
  db.transfer(req.session.userId,
    to, amount);
  res.json({ ok: true });
});
// Any site can POST here using victim's cookie
✓ Secure — CSRF Token Validation
// Express.js with csurf middleware
const csrf = require('csurf');
app.use(csrf({ cookie: true }));

app.post('/transfer', (req, res) => {
  // csurf auto-validates req.body._csrf
  // throws 403 if missing or wrong
  const { to, amount } = req.body;
  db.transfer(req.session.userId, to, amount);
  res.json({ ok: true });
});

DETECTION INDICATORS

What to look for in logs:

POST requests with unexpected Referer domains
Missing or mismatched Origin headers on state changes
Absent or invalid CSRF tokens in form submissions
Unusual activity immediately after user opens email links
State changes (transfers, PW resets) user didn't initiate
