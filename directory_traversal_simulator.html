WHAT IS DIRECTORY TRAVERSAL?

Directory Traversal (also called Path Traversal) occurs when an application uses user-controlled input to construct a file path and fails to restrict where that path can point. By inserting ../ sequences, an attacker climbs out of the intended directory and accesses arbitrary files anywhere on the server.

The attack does not require authentication or code execution. It is purely a file-read vulnerability — but the files it can expose (credentials, private keys, source code, config files) often lead directly to full system compromise.

Also known as: Path Traversal, "dot-dot-slash" attack, directory climbing. Classified under OWASP A01 — Broken Access Control.
user input: ../../../etc/passwd
→
server reads /etc/passwd
THE VULNERABLE SERVER STRUCTURE

A web application serves files from /var/www/html/files/. Here's what's on disk:

/ (filesystem root) ├── etc/ │ ├── passwd ← all OS users + shells │ ├── shadow ← hashed passwords (root only) │ └── ssh/ │ └── id_rsa ← private SSH key ├── home/ │ └── deployer/ │ └── .env ← API keys, DB passwords ├── var/ │ └── www/ │ └── html/ ← WEB ROOT (safe zone) │ ├── files/ │ │ ├── report.pdf │ │ └── brochure.pdf │ └── config.php ← DB credentials in source └── proc/ └── self/environ ← environment variables
The web server is only supposed to serve files inside /var/www/html/files/. Everything else is out of scope — unless traversal is possible.
HOW ../ TRAVERSAL WORKS

The application builds a file path like this:

// PHP — vulnerable file reader
$file = $_GET['filename'];
$path = "/var/www/html/files/" . $file;
readfile($path);
// No validation! Any path accepted.
Intended: /var/www/html/files/report.pdf
Traversal: /var/www/html/files/../../../etc/passwd
Resolves: /etc/passwd ← escapes web root!
Each ../ moves one directory up. Three of them take you from /var/www/html/files/ all the way to the filesystem root /.

BYPASSING WEAK DEFENSES

Developers often add naive filters to block traversal — checking for ../ in input. However, there are many ways to represent the same path that bypass these simple checks. Click each technique to see how it evades the filter and still reaches /etc/passwd.

Target: A filter that strips literal ../ from input — but nothing else.
1. Absolute Path (no traversal needed)
Payload: /etc/passwd    — skip ../ entirely if filter only checks for it
2. URL Encoding
Payload: ..%2F..%2F..%2Fetc%2Fpasswd    — %2F = / after URL decode
3. Double URL Encoding
Payload: ..%252F..%252F..%252Fetc%252Fpasswd    — %25 decodes to %, leaving %2F
4. Non-Standard Separators (Windows)
Payload: ..\..\..\Windows\win.ini    — backslash works on Windows servers
5. Strip-and-Reconstruct (Nested Traversal)
Payload: ....//....//....//etc/passwd    — filter removes ../ leaving ../
6. Null Byte Injection (legacy PHP)
Payload: ../../../etc/passwd%00.pdf    — %00 terminates string, .pdf suffix ignored

HOW TO PREVENT DIRECTORY TRAVERSAL
1. Canonicalize & Validate Paths
Resolve the full, canonical (absolute) path of the requested file before opening it. Then verify that the resolved path starts with the allowed base directory. This defeats all encoding tricks — the OS always resolves the real path first.
2. Allowlist Filenames
Instead of trying to block bad input, only permit a defined list of safe filenames or a strict pattern (e.g. /^[a-z0-9_\-]+\.pdf$/i). Reject anything that doesn't match — no traversal sequence can sneak through a strict allowlist.
3. Avoid User-Supplied Paths
Redesign the feature to use an indirect reference — e.g., a numeric file ID that maps to a server-side filename. The user never controls any part of the path. No traversal is possible when user input never touches the filesystem call.
4. Chroot / Containerization
Run the web server process inside a chroot jail or container where the filesystem root is the web root. Even if traversal succeeds, the attacker cannot escape the sandbox — there is no /etc/passwd to reach.
5. Least Privilege on Files
The web server process should only have read access to files it needs to serve. Sensitive files like /etc/shadow, .env, and private keys should have OS-level permissions that prevent the web server user from reading them.
6. WAF Rules
Web Application Firewalls can detect common traversal signatures (../, URL-encoded variants, null bytes). Not a primary defense — bypass techniques exist — but useful as a detection layer that generates alerts for incident response.
VULNERABLE vs SECURE CODE
❌ Vulnerable — No Path Validation
// PHP — direct path concatenation
$file = $_GET['filename'];
$path = '/var/www/html/files/' . $file;
readfile($path);

// Input: ../../../etc/passwd
// Opens: /etc/passwd — full read!
✓ Secure — Canonicalize & Prefix Check
// PHP — resolve and validate
$base = realpath('/var/www/html/files');
$file = $_GET['filename'];
$path = realpath($base . '/' . $file);

if ($path === false ||
    strpos($path, $base) !== 0) {
  http_response_code(403);
  exit('Access denied');
}
readfile($path);
// realpath() resolves ../ before check
❌ Vulnerable — Naive Strip Filter
// Node.js — strips ../ but bypassable
const file = req.query.file
  .replace(/\.\.\//g, '');
const path = '/var/www/files/' + file;
fs.readFile(path, ...);

// Input: ....//....//etc/passwd
// After strip: ../../etc/passwd — still traverses!
✓ Secure — Indirect File Reference
// Node.js — ID lookup, no path from user
const FILE_MAP = {
  '1': 'report.pdf',
  '2': 'brochure.pdf',
};
const id = req.query.id;
const filename = FILE_MAP[id];
if (!filename) return res.status(404);
const safe = path.join(BASE, filename);
fs.readFile(safe, ...);
// User never controls any path component
DETECTION INDICATORS

What to look for in logs:

../ or ..\ in URL parameters or headers
URL-encoded variants: %2F %2E %252F
References to sensitive paths: /etc/passwd win.ini
Repeated 403/404 errors with path-like filenames
Requests for files outside the web root directory
