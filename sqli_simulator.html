WHAT IS SQL INJECTION?

SQL Injection (SQLi) occurs when an attacker inserts — or "injects" — malicious SQL code into a query that an application sends to its database. If the application doesn't properly validate or sanitize user input, the database executes the attacker's code alongside the legitimate query.

Because web applications typically build queries by concatenating strings with user input, an attacker can break out of the intended string and append arbitrary SQL. The database cannot distinguish the injected code from the developer's original query — it sees valid SQL and obeys it.

Real impact: Unauthorized login, full database dump, credential theft, data destruction, privilege escalation, and in some cases, operating system command execution.
THE VULNERABLE QUERY

A login form that builds a query by string concatenation:

SELECT * FROM users WHERE username = '[USERNAME]' AND password = '[PASSWORD]';

What the developer intended:

SELECT * FROM users WHERE username = 'alice' AND password = 'mypassword';

What an attacker sends instead:

SELECT * FROM users WHERE username = 'admin' OR '1'='1' AND password = 'anything'; -- '1'='1' always true → authentication bypassed!

ATTACK LAB — UNION-BASED DATA DUMP

BLIND SQLi — TIME-BASED

When the application doesn't return query results, attackers use time-based blind injection. The payload makes the database sleep for a set number of seconds when a condition is true. By observing response delay, the attacker can extract data one bit at a time.

Technique: IF(condition, SLEEP(5), 0) — if the database name starts with 'b', wait 5 seconds, else return immediately.

HOW TO PREVENT SQL INJECTION
1. Parameterized Queries
The #1 defense. Separate the query structure from the data. The database driver handles quoting and escaping automatically — user input can never change query logic.
2. Stored Procedures
Precompiled SQL stored in the database. When parameters are bound correctly, input is always treated as data — not executable code — by the query engine.
3. Input Validation
Enforce strict allowlists on input type, length, and format. Reject or sanitize characters with special SQL meaning: ' -- ; UNION SELECT. Never rely on this alone.
4. Least Privilege
The database account used by the application should only have the permissions it needs. A login form should never use an account with DROP, CREATE, or GRANT rights.
5. Web Application Firewall
A WAF can detect common SQLi signatures and block malicious requests before they reach the application layer. Not a substitute for secure coding — but adds a useful detection layer.
6. Error Handling
Never expose raw database errors to users. SQL error messages reveal table names, column names, and database version — information attackers use to craft precise payloads.
VULNERABLE vs SECURE CODE
❌ Vulnerable — String Concatenation
// PHP example — directly embeds user input
$user = $_POST['username'];
$pass = $_POST['password'];

$query = "SELECT * FROM users
 WHERE username = '" . $user . "'
 AND password = '" . $pass . "'";

// Input: admin' -- → password check skipped!
✓ Secure — Parameterized Query
// PHP PDO example — input bound as data
$stmt = $pdo->prepare(
 "SELECT * FROM users
 WHERE username = ?
 AND password = ?"
);

$stmt->execute([$user, $pass]);
// Injection impossible — ? is always literal data
❌ Vulnerable — Raw ORM Query
// Node.js — raw SQL with template literals
const q = `SELECT * FROM users
 WHERE id = ${req.params.id}`;
await db.query(q);

// Input: 1 UNION SELECT * FROM users -- → dump!
✓ Secure — Parameterized ORM
// Node.js — parameterized binding
const q = "SELECT * FROM users
 WHERE id = $1";
await db.query(q, [req.params.id]);

// $1 is a placeholder — input treated as value only
DETECTION INDICATORS

What to look for in logs:

Unusual characters in parameters: ' " -- ; UNION SELECT
Repeated requests with slight input variations
DB error messages in HTTP responses
Anomalous query execution times (blind SQLi)
Requests to information_schema tables
