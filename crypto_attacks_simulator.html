What Is a Downgrade Attack?
A downgrade attack forces a connection to use an older, weaker version of a protocol or cipher suite than both parties actually support. The attacker acts as a man-in-the-middle, intercepting the handshake and stripping out the strong options — leaving only the breakable ones.

The server and client don't realize they've been downgraded. They complete the handshake believing they've negotiated normally.
Exam anchor: "POODLE", "BEAST", "DROWN", "FREAK" are all named downgrade attacks. Any question mentioning "forced to use an older protocol" or "SSL instead of TLS" → Downgrade Attack.
THE CIPHER SUITE NEGOTIATION
During a TLS handshake, the client sends a list of cipher suites it supports — from strongest to weakest. The server picks the best one it also supports. A downgrade attack removes the strong suites in transit, forcing selection of a breakable one.

How to Stop Downgrade Attacks
Disable Legacy Protocols
Disable SSL 2.0, SSL 3.0, TLS 1.0, TLS 1.1 entirely on servers. If weak versions aren't available, they can't be negotiated. This is the simplest and most complete defense.
TLS Fallback SCSV
A special cipher suite value (TLS_FALLBACK_SCSV) tells the server "the client is using a downgraded version on purpose." If the server supports higher versions, it aborts — making the MITM visible.
HSTS
HTTP Strict Transport Security tells browsers to always use the highest TLS available and refuse protocol downgrades. Enforced at the browser level — attackers can't negotiate down even if they intercept.
What Is a Collision Attack?
A collision attack finds two different inputs that produce the same hash output. Hash functions are supposed to be a unique fingerprint — a collision breaks that guarantee.

If an attacker can find a collision, they can substitute one document for another while keeping the same hash — making a forged file appear authentic.
Exam anchor: "Two different inputs with the same hash" = Collision Attack. MD5 and SHA-1 are broken by real collision attacks. SHA-256+ is still safe. When asked about the "result" of finding two matching hashes — that is the collision.

Real-World Impact + Defenses
Real Collision Attacks
MD5 — BROKEN (1996)
Researchers produced two different X.509 certificates with the same MD5 hash. Used to forge a rogue CA certificate. Never use MD5 for integrity.
SHA-1 — BROKEN (SHAttered, 2017)
Google's Project Zero produced two different PDF files with the same SHA-1 hash. SHA-1 is deprecated for code signing and certificates.
SHA-256 / SHA-3 — SAFE (so far)
No known practical collisions. The output space (2²⁵⁶ possibilities) is too large for current computing to exhaust, even with quantum considerations.
Defenses
Use Strong Hash Functions
SHA-256, SHA-384, SHA-512, SHA-3. The larger the output space, the harder the collision. Avoid MD5 and SHA-1 in any security context.
HMAC (Keyed Hash)
Adds a secret key to the hash. An attacker finding a collision can't forge a valid HMAC without the key — even if they find a collision in the underlying hash.
Digital Signatures
Code signing, certificate chains, and document signatures over strong hashes prevent substitution attacks — any modification changes the signature.
What Is a Birthday Attack?
A birthday attack is the method used to find collisions — it exploits the birthday paradox in probability. It seems counterintuitive: you need far fewer attempts than expected to find a matching pair.

The birthday problem: in a room of just 23 people, there's a 50% chance two share a birthday. Not 183 people (half of 365). The math of pairs grows much faster than the space.

For a hash with an N-bit output, brute force needs 2ᴺ attempts. A birthday attack needs only 2^(N/2) — the square root. A 64-bit hash takes only 2³² attempts (~4 billion) to find a collision, not 2⁶⁴.
Exam anchor — the critical distinction: Birthday attack = the probability method (square root of the space). Collision attack = the result (two inputs with the same hash). On an exam: "Birthday attack is used to find collisions." They work together.
